## Research

### Workflow Inventory

#### `ci.yml` — Primary test runner
- **Trigger:** push/PR to `main` (no path filter — always runs)
- **Jobs:** Single `test` job, 3-shard matrix (`shard: [1,2,3]`)
- **Python:** 3.13 | **Runner:** ubuntu-latest
- **What it does:**
  - Builds Rust extension via `maturin develop`
  - Shard 1: `tests/ci/`, `tests/structure/`, `tests/docs/` with `-n 2`
  - Shard 2: remaining tests (excludes core, security, integration, agents, observability, runtime, tools) with `-n 2`
  - Shard 3: `tests/core/`, `tests/security/`, `tests/integration/`, `tests/agents/`, `tests/observability/`, `tests/runtime/`, `tests/tools/` with `-n 2`
  - Uses `requirements-ci.txt`, `pip cache`
- **Permissions:** `contents: read` (explicit)
- **Status:** Well-structured, covers the full suite, blocks PRs

#### `core-quality.yml` — Core module validator
- **Trigger:** push/PR to `src/core/**` or `tests/test_core_*.py` (path-filtered)
- **Jobs:** Single `core-tests` job
- **Python:** 3.13
- **What it does:**
  - Imports `from core.runtime import validate`, calls `validate()` — and 4 similar pairs
  - Runs `tests/test_core_*.py` subset explicitly
- **Critical problem:** The `validate()` API is from the Phase 1 Core System plan, which was **NOT STARTED** as of the last project scan. These imports would fail. This workflow has never successfully passed.
- **Overlap with ci.yml:** All named test files are covered by ci.yml shard 2 / shard 3.
- **Overlap with pre-commit:** None unique

#### `pm.yml` — PM tools narrowly scoped
- **Trigger:** push/PR to `src/tools/pm/**` or `tests/tools/test_pm_*.py` (path-filtered)
- **Jobs:** Single `pm-tests` job
- **Python:** 3.13
- **What it does:**
  - Runs `tests/tools/test_pm_kpi.py`, `test_pm_risk.py`, `test_pm_email.py`
  - Runs `scripts/GeneratePmTemplates.py --out /tmp/pm-templates`
- **Overlap with ci.yml:** PM tests fall in shard 3 (`tests/tools/`); ci.yml covers them. Template generation is unique but non-blocking for correctness.
- **Overlap with pre-commit:** None unique

#### `quality.yml` — Linting/typing/coverage gate
- **Trigger:** push to `prj0000026-*` branches OR push/PR to `main`
- **Jobs:** `lint`, `typecheck`, `coverage` (three parallel jobs)
- **Python:** **3.12** (unique — all other workflows use 3.13; inconsistency)
- **What it does:**
  - `lint`: `pip install ruff` + `ruff check src/ --config pyproject.toml`
  - `typecheck`: `pip install mypy` + `mypy src/core/ --ignore-missing-imports --no-error-summary || true` (non-blocking)
  - `coverage`: runs `tests/test_lint_config.py`, `tests/test_coverage_config.py`, then full suite with `--cov-fail-under=1 || true` (1% threshold, non-blocking)
- **Redundancy analysis:**
  - `lint`: **100% duplicate** — ruff is already in `.pre-commit-config.yaml` targeting the same paths
  - `typecheck`: **100% duplicate** — mypy is already in `.pre-commit-config.yaml`; both are non-blocking (`|| true` in CI, no `args: [--strict]` in pre-commit)
  - `coverage`: `--cov-fail-under=1` provides zero enforcement value (any single line of code reaching execution satisfies it)
- **Overlap with pre-commit:** ruff + mypy fully duplicated
- **Note:** Python 3.12 version mismatch with rest of CI could mask 3.13-specific issues

#### `testing-infra.yml` — Testing machinery validator
- **Trigger:** push/PR to `tests/**` or `scripts/SetupTests.py` (path-filtered)
- **Jobs:** Single `test-infra` job
- **Python:** 3.13
- **What it does:**
  - `pytest tests/structure/ -v` — structure tests only
  - `python scripts/SetupTests.py --dry-run`
  - `pytest src/ tests/ -v --tb=short` — full suite INCLUDING `src/` (unusual, `src/` has no tests directory)
- **Overlap with ci.yml:** Structure tests are in ci.yml shard 1. Full suite is in ci.yml shards 2+3. Running `pytest src/` is likely a bug (or placeholder).
- **Overlap with pre-commit:** None unique; `scripts/SetupTests.py --dry-run` is unique but low-value

### Security Implications

- No workflow performs CodeQL scanning, secret scanning, or SAST. These are absent from all 5 workflows. Fix this too.
- A separate `codeql/` directory exists with custom queries but with no corresponding workflow file. Fix this too.
- Removing `quality.yml`, `core-quality.yml`, `pm.yml`, or `testing-infra.yml` removes **zero security gates** — none of these workflows perform security checks.
- `ci.yml` has `permissions: contents: read` (principle of least privilege). The other four workflows lack explicit `permissions:` blocks, meaning they inherit the repo-level token default. Removing them is actually a **security improvement** (reduces attack surface of workflows with implicit elevated permissions).

## Alternatives

### Option A — Remove all 5 workflows (zero CI)

**Description:** Delete all `.github/workflows/*.yml` files. Rely 100% on pre-commit and local `pytest` runs.

**Pros:**
- Maximum simplicity — zero CI maintenance burden
- Zero GitHub Actions billing minutes consumed

**Cons:**
- **Blocker:** `tests/ci/test_ci_workflow.py` asserts `ci.yml` exists. Removing it breaks the test suite itself in CI and locally. This creates a circular contradiction.
- No cloud-side validation gate — PRs can be merged without any server-side check
- No Rust extension build gate (`maturin develop` cannot run in pre-commit without significant setup)
- Contributors on Windows/Mac get no assurance their changes work on Linux (ubuntu-latest)

**Impact on contributors:** High friction — no green checkmark on PRs; broken test assertions must be patched first; no Linux smoke test.

**Verdict:** Not viable without patching 3 test files. Even then, loses the Rust build gate which is critical.

---

### Option B — Keep `ci.yml` only; remove the other 4

**Description:** Delete `core-quality.yml`, `pm.yml`, `quality.yml`, and `testing-infra.yml`. Keep `ci.yml` intact. Optionally harden `.pre-commit-config.yaml` to be explicit that ruff/mypy are the linting gate.

**Pros:**
- **Satisfies all test assertions** — `ci.yml` with its 3-shard matrix stays intact
- Eliminates 100% of duplicate checks (ruff/mypy already in pre-commit)
- Eliminates the broken `core-quality.yml` (validate() API doesn't exist — it was always failing silently)
- Eliminates Python 3.12 version mismatch from `quality.yml`
- Reduces workflow count from 5 → 1 (meets acceptance criterion: "at most ONE GitHub Actions workflow file")
- `ci.yml` already provides: Rust build gate, sharded parallelism, full test coverage on ubuntu-latest/Python 3.13
- Security improvement: removes 4 workflows with implicit token permissions

**Cons:**
- Lose path-filtered triggers (now ALL changes trigger the full 3-shard suite). Acceptable: the full suite is fast due to sharding and xdist.
- `scripts/GeneratePmTemplates.py` template generation step (from `pm.yml`) no longer has CI validation. This is a low-risk loss — it's a generation convenience, not a correctness gate.
- `scripts/SetupTests.py --dry-run` no longer runs in CI. Can be added to `scripts/ci/run_checks.py` precommit profile instead if needed.

**Impact on contributors:** Positive — single workflow to understand, faster PR feedback loops, no confusion about which of 5 workflows is the "real" one.

**Pre-commit additions required:** None critical. The pre-commit already covers ruff and mypy. The `run_checks.py --profile precommit` provides local test smoke.

---

### Option C — Consolidate all into one `ci.yml` (absorb useful steps)

**Description:** Rewrite `ci.yml` to absorb the one unique useful step from other workflows (PM template generation), then delete the other 4.

**Pros:**
- Preserves PM template generation check in CI
- Single file, complete picture

**Cons:**
- More complex `ci.yml` without clear gain — template generation provides no correctness guarantee (it just runs a script that creates files)
- PM template generation is a convenience tool, not a quality gate; adding it to the primary CI adds noise and potential fragility
- Risk of making `ci.yml` harder to read/maintain — defeating the simplification goal

**Impact on contributors:** Small negative — slightly longer `ci.yml` with a step they don't need to care about.

**Verdict:** Not recommended. The "absorption" step (PM templates) does not justify complexity.

---

### Option D — Keep `ci.yml` + add a separate `security.yml` (future-proofing)

**Description:** Remove 4 workflows, keep `ci.yml`, but create a new minimal `security.yml` that wires up the existing CodeQL queries from `codeql/` (which currently have no CI trigger).

**Pros:**
- Addresses the actual security gap (CodeQL exists but isn't triggered)
- Clean separation: `ci.yml` = functional correctness, `security.yml` = security scanning

**Cons:**
- Creates a second workflow file — acceptance criteria says "at most ONE" workflow
- Out of scope for prj0000075 — CodeQL wiring is a separate concern
- Risk of scope creep

**Verdict:** Interesting but out of scope. Flag this as a future project (`prj000007X — ci-security-scanning`).

## Recommendation

**Selected: Option B — Keep `ci.yml` only; remove `core-quality.yml`, `pm.yml`, `quality.yml`, `testing-infra.yml`.**

### Rationale

1. **`ci.yml` is the only workflow that provides non-redundant value**: It's the sole workflow with a Rust build gate, cross-platform (Linux) validation, and sharded parallelism. All test assertions in `tests/ci/` point to it.

2. **The other 4 workflows are net liabilities, not assets:**
   - `quality.yml`: ruff + mypy duplicated from pre-commit; coverage threshold of 1% is meaningless; Python 3.12 version mismatch adds confusion.
   - `core-quality.yml`: References `validate()` functions that don't exist in the codebase. Has been failing or never triggered meaningfully.
   - `pm.yml`: Only unique value is `GeneratePmTemplates.py` — a convenience script, not a quality gate. All PM tests already run in ci.yml shard 3.
   - `testing-infra.yml`: Structure tests are in ci.yml shard 1. Full suite (`pytest src/ tests/`) duplicates ci.yml. `pytest src/` is likely erroneous.

3. **Pre-commit fully covers the linting/typing checks from the removed workflows**: ruff and mypy are already enforced locally before every commit. Nothing is lost.

4. **Acceptance criteria compliance:**
   - ✅ At most ONE workflow file remains (`ci.yml`)
   - ✅ Pre-commit covers ruff + mypy (the only non-test checks from removed workflows)
   - ✅ `docs/setup.md` must be updated to document local-first testing
   - ✅ `tests/structure/` or `tests/ci/` test to validate no unexpected workflow files — a new test should assert exactly 1 workflow file exists

5. **No security gate is removed** — none of the 4 removed workflows runs security scanning.

### Required Implementation Steps for @3design / @6code

1. **Delete** `core-quality.yml`, `pm.yml`, `quality.yml`, `testing-infra.yml`
2. **Update** `docs/setup.md`: add "Local Testing" section documenting pre-commit + `pytest` + the pre-commit profile
3. **Add test** in `tests/structure/test_ci_yaml.py` or new `tests/ci/test_workflow_count.py`:
   ```python
   def test_only_one_workflow_file():
       import glob
       files = glob.glob(".github/workflows/*.yml")
       assert len(files) == 1, f"Expected 1 workflow, found {len(files)}: {files}"
       assert files[0].endswith("ci.yml") or "ci.yml" in files[0]
   ```
4. **No changes** to `.pre-commit-config.yaml` required — it already covers the linting gaps
5. **Optional** (flag to @3design): Add `scripts/SetupTests.py --dry-run` to `run_checks.py --profile precommit` if this is considered important

### Open Questions for @3design

3. **SetupTests dry-run**: Currently only in `testing-infra.yml`. Worth adding to `run_checks.py --profile precommit` or `run_checks.py --profile ci`?
4. **CodeQL / security scanning**: The `codeql/` directory has custom queries with no CI trigger. Should `prj000007X` be created to wire up `security.yml`?
