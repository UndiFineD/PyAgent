## 1. Workflow injection analysis — `security.yml`

### 1a. User-controlled data in `run:` steps

No `run:` steps exist in `security.yml`. All steps use `uses:` (actions), not inline shell
commands. There is no pathway for user-controlled data (e.g., `github.event.pull_request.title`,
`github.event.issue.body`, `github.head_ref`) to reach a shell execution context.

**Result: SAFE — no injection vector exists.**

### 1b. Matrix expansion

The only `${{ }}` expression injected into a step input is:

```yaml
languages: ${{ matrix.language }}
category: "/language:${{ matrix.language }}"
```

`matrix.language` originates from the static definition `[python, javascript]` in the workflow
file itself — not from any external event context. It cannot be overridden by a PR author or
external actor.

**Result: SAFE — static matrix values only.**

### 1c. Trigger: `pull_request` vs `pull_request_target`

The workflow uses `pull_request` (not `pull_request_target`). Under `pull_request`, fork PR runs
execute in the **fork's isolated context** with read-only permissions to the base repository
secrets. The `secrets:` context is unavailable to fork PRs.

Under `pull_request_target`, the workflow would run in the base repo context and have access to
secrets — a known high-risk pattern when combined with code checkout from the PR branch.

**Result: SAFE — correct trigger used. No secret leakage to fork PRs.**

### 1d. Permissions

```yaml
permissions:
  contents: read          # Checkout only
  security-events: write  # Upload SARIF to Security tab
  actions: read           # Required by CodeQL init on private repos
```

These are minimal and exactly match the permissions required by the CodeQL action. No `write`
access to `pull-requests`, `packages`, `id-token`, or other sensitive scopes is granted.

The default GitHub Actions permission model (when no `permissions:` block is set) grants
`contents: write` and potentially more, depending on repo settings. The four deleted workflows
(`core-quality.yml`, `pm.yml`, `quality.yml`, `testing-infra.yml`) had **no `permissions:`
block** and therefore inherited the repo default. Their removal is a **net security improvement**.

**Result: SAFE — minimal permissions declared; removal of unpermissioned workflows is hardening.**

## 2. Action pinning

| Action | Pin strategy | Risk |
|--------|-------------|------|
| `actions/checkout@v4` | Version tag | LOW — GitHub-owned action, widely audited |
| `github/codeql-action/init@v3` | Version tag | LOW — GitHub-owned action |
| `github/codeql-action/autobuild@v3` | Version tag | LOW — GitHub-owned action |
| `github/codeql-action/analyze@v3` | Version tag | LOW — GitHub-owned action |

All actions are GitHub-owned and pinned to major version tags rather than full SHAs. SHA pinning
(`@abc123def...`) would prevent a compromised `@v3` tag from delivering malicious code. However,
for GitHub-owned actions this risk is exceptionally low, and SHA pinning dramatically increases
maintenance burden without proportional security gain at this project scale.

**Recommendation (INFORMATIONAL — not blocking):** If the project adopts Dependabot for Actions
updates (`.github/dependabot.yml`), SHA pinning becomes practical and is worth revisiting then.
A future project can address this; it is out of scope for prj0000075.
