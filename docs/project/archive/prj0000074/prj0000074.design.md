## Design 2 — Docstrings Policy

**Decision:** Add a `## Docstrings policy` section to `5test.agent.md` and `6code.agent.md` after the agent's scope table.

**Content of the section:**
```
## Docstrings policy

Every function, method, and class written or modified by this agent MUST include a Google-style
docstring. This is a hard requirement — code without docstrings will not be accepted.

Enforcement:
- ruff rules D100–D107 (missing docstrings), D200–D215 (formatting)
- Run: `ruff check --select D <file>` to verify

Format:
  def my_func(x: int) -> str:
      """One-line summary.

      Args:
          x: Description of x.

      Returns:
          Description of return value.
      """
```

## Design 5 — Deeper @2think

**Decision:** Insert between Step 2 and Step 3 a `## Research task types` section listing exactly 6 task types, and insert a `## Minimum depth guideline` before Step 1.

**6 research task types:**
1. **Literature review** — search docs, README, prior project artifacts for established patterns
2. **Alternative enumeration** — list all viable approaches, not just the obvious one
3. **Prior-art search** — check `docs/architecture/archive/` and closed project folders for related prior decisions
4. **Constraint mapping** — enumerate technical, time, scope, and team constraints that bound the solution space
5. **Stakeholder impact** — identify which agents, users, or system components are affected by each option
6. **Risk enumeration** — for each option, list minimum 3 failure modes and their likelihood/impact

**Minimum depth guideline:**
- Generate ≥ 3 distinct options (not variations on the same theme)
- Research each option across ≥ 4 of the 6 task types above before scoring
- Evidence must cite at least one file path from the workspace per option
- Do not skip to the recommendation if fewer than 2 options have been fully analysed
