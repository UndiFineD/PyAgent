## Problem Analysis

### Root Cause

The original `useWebSocket` hook used a **fixed reconnect delay** (default
`reconnectDelay: 3000 ms`).  When many browser tabs reconnect simultaneously
after a backend restart they form a **thundering-herd storm**: all tabs hit the
backend at the same instant, often causing a cascade of connection failures.

### Why exponential backoff + jitter?

| Strategy | Behaviour | Thundering-herd risk |
|---|---|---|
| Fixed delay (old) | All reconnects at `t + 3 s` | High — synchronized storm |
| Exponential only | Delays grow: 1 s, 2 s, 4 s … | Medium — still synchronized |
| Exponential + jitter | Random spread in `[0.5·exp, exp]` | Low — desynchronized |

Full-jitter is the standard AWS recommendation for retries in distributed
systems (Exponential Backoff And Jitter, AWS blog 2015).

### Alternative approaches considered

1. **Token-bucket / rate-limiter on the server** — shifts complexity to the
   backend.  Does not help when the backend is genuinely down; only reduces
   spike on restart.  Rejected as out-of-scope for a pure-frontend hook.

2. **Fixed delay with ±jitter** — cheap but does not reduce the number of
   concurrent connections over time; tabs still retry at similar frequencies.
   Rejected in favour of exponential growth.

3. **Server-Sent Events (SSE)** — one-way, browser handles reconnects natively.
   Does not support bidirectional messaging (send / receive).  Unsuitable for
   agent task submissions.

4. **WebSocket multiplexing library (e.g., socket.io)** — adds a heavyweight
   dependency; protocol upgrade; breaks existing message framing.  Rejected.

### Chosen solution

- Replace the single `reconnectDelay` option with three composable options:
  `baseDelay` (default 1 000 ms), `maxDelay` (default 30 000 ms), `maxRetries`
  (default `Infinity`).
- Backoff formula: `exp = min(maxDelay, baseDelay × 2^attempt)`.
- Jitter application: `jittered = exp × (0.5 + random() × 0.5)`.
- Expose `reconnectAttempts` in the hook return so UIs can show progress.
- Keep `reconnectDelay` as a deprecated alias for `baseDelay` — zero breaking
  changes for existing consumers.

### Risk assessment

| Risk | Likelihood | Mitigation |
|---|---|---|
| Backwards-compat regression | Low | `reconnectDelay` kept as alias; tests confirmed |
| Timer leak on unmount | Medium (pre-existing) | `active` flag + `clearTimeout` on cleanup |
| Infinite retry consuming resources | Low | `maxRetries` option; documented default is ∞ |
| Jitter producing delays of 0 ms | Impossible | lower bound is `0.5 × exp ≥ 500 ms at attempt 0` |

---

## Conclusion

Exponential backoff with full jitter is the correct and well-validated approach.
The change is confined to `web/hooks/useWebSocket.ts` and is non-breaking for
existing consumers.  Implementation risk is low.
