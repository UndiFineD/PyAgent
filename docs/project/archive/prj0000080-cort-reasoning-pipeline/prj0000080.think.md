## 1. Root Cause Analysis

PyAgent's current agents issue a **single prompt–response cycle**: one prompt in, one
response out. There is no mechanism for:

1. **Self-critique** — the agent cannot evaluate its own output for correctness or
   completeness before returning it to the caller.
2. **Diversity of reasoning** — a single temperature setting produces a single solution
   trajectory; optimal answers often require exploring multiple solution paths.
3. **Iterative refinement** — without feedback loops, systematic errors (incorrect
   assumptions, incomplete coverage, shallow reasoning) propagate directly to callers.

Chain-of-Recursive-Thoughts addresses all three by wrapping any LLM call in a
multi-round loop that generates `M` alternatives per round, scores each against a
rubric, and uses the best candidate as the seed for the next round. The project goal
is to build this as a **reusable async mixin** so any `BaseAgent` subclass can opt
into CoRT reasoning with minimal code changes.

The specific gap this project fills:

| Current State | Target State |
|---|---|
| Single LLM call per agent task | N rounds × M alternatives per task |
| No answer scoring | Scored on correctness, completeness, reasoning-depth |
| No self-critique loop | Each round refines the prior best answer |
| Temperature fixed at agent level | Per-alternative temperature variation |
| No recursion tracking in reasoning | ContextTransaction guards against infinite re-entry |

## 2. CoRT Algorithm Fundamentals

### 2.1 The Core Algorithm

Chain-of-Recursive-Thoughts (CoRT) is a meta-reasoning strategy that wraps any
generative LLM call in a **multi-round tournament loop**:

```
Round 1:
  Prompt → generate M alternative chains (at temperatures T1…Tm)
  Score each chain against rubric → select best chain C*

Round 2:
  C* becomes context → generate M new alternatives (each critiques and improves C*)
  Score → select best chain C**

Round N:
  Repeat → final winner returned as agent response
```

Key parameters:
| Parameter | Default | Description |
|---|---|---|
| `rounds` (N) | 3 | Number of recursive self-critique iterations |
| `alternatives` (M) | 3 | Competing chains generated per round |
| `base_temp` | 0.7 | Temperature of the first alternative (closest to greedy) |
| `temp_step` | 0.1 | Increment per alternative to increase diversity |
| `max_temp` | 1.2 | Hard ceiling on any single alternative temperature |
| `score_threshold` | 0.05 | Early-stop delta (used only in adaptive variant) |

Temperature schedule for M=3: `[0.7, 0.8, 0.9]`
Temperature schedule for M=5: `[0.7, 0.8, 0.9, 1.0, 1.1]`

### 2.2 How CoRT Differs from Related Techniques

| Technique | Branching | Rounds | Self-evaluation | Key distinction |
|---|---|---|---|---|
| **Chain-of-Thought (CoT)** | None | 1 | None | Step-by-step reasoning, single path |
| **Self-Refine** | None | N | Simple heuristic | Re-prompts model to critique own last output; serial, no competition |
| **Reflexion** | None | N | Episodic memory | Stores reflection traces across tasks; broader in scope |
| **Tree-of-Thoughts (ToT)** | Branching tree | N | BFS/DFS search | Explores branching search tree; much higher cost; designed for complex multi-step problems |
| **Constitution AI / RLHF** | None | 2 | Model critique | Offline fine-tuning oriented; not an inference-time loop |
| **CoRT (this project)** | Flat M per round | N | Rubric + judge | Fixed-width, N rounds × M alternatives, scored competition |

The critical distinction: CoRT is **flat-competitive** rather than tree-branching.
Each round generates exactly M alternatives in parallel and selects exactly one winner.
This bounds cost to `O(N × M)` LLM calls, whereas Tree-of-Thoughts can grow
exponentially with branching factor.

### 2.3 Evaluation Rubric Axes (per project spec)

| Axis | Weight (default) | Description |
|---|---|---|
| Correctness | 0.5 | Factual accuracy, logical consistency, no contradictions |
| Completeness | 0.3 | Covers all aspects stated in the original prompt |
| Reasoning-depth | 0.2 | Step-by-step derivation shown, not a bare conclusion |

**Composite score** = 0.5 × correctness + 0.3 × completeness + 0.2 × reasoning_depth
where each axis is 0–10. Final score is therefore in range [0, 10].

## 3. Prior Art in This Codebase

### 3.1 `src/cort/__init__.py` — Existing CoT Tree Module

An earlier `ChainOfThought` / `ThoughtNode` module exists at `src/cort/__init__.py`.
It provides:
- `ChainOfThought(context_mgr)` — root container with async `new_node(text)`
- `ThoughtNode` — branching node with async `fork(text)` and `add(text)`
- Context manager integration via an injected `context_manager.ContextManager` object

**Key observations:**
1. This module is a **tree structure** for recording thought branches — it does not
   implement any scoring, rubric evaluation, or recursive selection logic.
2. It depends on a `context_manager` module that does not exist in
   `src/transactions/` or `src/core/`; the import would fail at runtime.
3. It shares the `src/cort/` namespace with the proposed project, creating a
   **namespace collision risk** if `src/core/reasoning/` is the new target.
4. It does not integrate with `ContextTransactionManager.py` (uses a different,
   non-existent CM).

**Decision for this project:** The new `src/core/reasoning/` module is the correct
location (per project spec). The pre-existing `src/cort/` module is archived prior
art; do NOT inherit from or import it. Use `ContextTransactionManager.py` from
`src/transactions/` directly.

### 3.2 `src/transactions/ContextTransactionManager.py`

Provides `ContextTransaction` with:
- `context_id` based recursion guard (raises `RecursionGuardError` on re-entry)
- Per-asyncio-task isolation via `contextvars.ContextVar`
- Stack-based parent tracking (`parent_id` auto-populated from stack head)
- Both sync (`__enter__`/`__exit__`) and async (`__aenter__`/`__aexit__`) context managers

This is exactly the guard needed for CoRT. Each `CortCore.reason(prompt)` invocation
must open a `ContextTransaction` keyed on a stable invocation ID to prevent:
- CortCore recursively calling itself via a mixin agent
- Two concurrent CoRT tasks sharing the same context ID

### 3.3 `src/transactions/MemoryTransactionManager.py`

Provides `MemoryTransaction` with:
- Transactional key-value store with `commit()` / `rollback()`
- Both sync (RLock) and async (asyncio.Lock) context managers
- Optional remote sync endpoint

This is the store for caching reasoning chains between rounds (so the winning chain
from round N is the seed for round N+1) and for optional result caching across
identical prompts.

### 3.4 `src/agents/BaseAgent.py`

- Abstract `BaseAgent(ABC)` with `AgentManifest`, `AgentLifecycle`, asyncio semaphore
- `run(task)` as the abstract method
- No mixin pattern enforced; mixins in PyAgent are implemented via Python multiple
  inheritance (regular class, no `ABC` constraint)
- The existing `src/core/base/mixins/` path does not exist yet — it is a target
  described in the project instructions but not created

### 3.5 `src/core/providers/FlmChatAdapter.py`

- Wraps `openai.OpenAI` client (OpenAI-compatible API)
- Depends on `FlmProviderConfig` (dataclass with base_url, model, timeout, retries)
- Defines `_ClientProtocol`, `_ChatClient`, `_CompletionCreator` protocols
- `ToolExecutor = Callable[[_ToolCallProtocol], str]` for tool call handling

The LLM call interface is `client.chat.completions.create(**kwargs)` — standard
OpenAI-compatible. CortCore must accept a **callable** that matches this interface
to remain backend-agnostic.

## 5. Options: Recursive Loop Design

### Option A — Fixed N Rounds, Fixed M Alternatives

**Description:**
The simplest possible loop. Parameters `rounds` and `alternatives` are set at
construction and never change. Every round generates exactly M alternatives regardless
of convergence. All N rounds run unconditionally.

```
round 1: generate [a1, a2, a3]  → score → select best → seed
round 2: generate [a1, a2, a3]  → score → select best → seed
round 3: generate [a1, a2, a3]  → score → select best → return
```

**Total LLM calls:** N × M = 3 × 3 = 9 (for defaults), plus up to N × M scoring
calls if using LLM-as-judge.

**Implementation complexity:** Low. Single for-loop with no conditional branching on
scores. Trivially testable with mocked LLM calls.

| Dimension | Assessment |
|---|---|
| Quality | Consistent; not highest (wastes rounds if converged early) |
| Token cost | Fully predictable; always N×M calls |
| Latency | Fully predictable; N rounds sequential, M alternatives in parallel |
| Complexity | Very low; simple for-loop |
| Testability | High; deterministic call count, easy to mock |
| Risk | Low; no special edge cases |

**Pros:**
- Completely predictable token budget, easy to expose as `max_tokens_budget`
- Easiest to unit test; test expects exactly N*M calls
- No threshold tuning required
- Implementation matches the simplest correct description of CoRT

**Cons:**
- Wastes computation if answer converges after round 1 or 2
- No way to skip rounds if quality already excellent

---

### Option B — Adaptive Rounds (Early Stop on Score Delta < Threshold)

**Description:**
Same basic loop as Option A, but compares the current round's best score against the
previous round's best score. If the improvement (`delta = score_new - score_old`) is
below `score_threshold`, stop early and return the current best.

```
round 1: generate M → score → best_score=7.2 → delta=+inf → continue
round 2: generate M → score → best_score=7.8 → delta=0.6  → continue
round 3: generate M → score → best_score=7.9 → delta=0.1  → delta < 0.2? → STOP early
```

**Total LLM calls:** Variable, between `1 × M` and `N × M`. On average 1.5–2 rounds
for typical structured questions (empirical: answers converge quickly for well-formed
prompts, CoRT paper shows ~70% convergence by round 2 with threshold 0.05).

**Implementation complexity:** Medium. Requires score tracking across rounds, threshold
parameter, and an early-exit conditional. Still a simple loop but with state.

| Dimension | Assessment |
|---|---|
| Quality | Slightly higher than A on average (stops when gains marginal); same max |
| Token cost | Lower on average; min cost 1×M, max N×M |
| Latency | Lower on average; earlier stop = fewer sequential rounds |
| Complexity | Low-medium; minimal extra state (one float) |
| Testability | Medium; must test early-exit branch and threshold boundary |
| Risk | Low-medium; threshold tuning required, wrong threshold wastes attempts or exits too early |

**Pros:**
- Respects token budget while extracting full quality when needed
- Reduces average latency without sacrificing maximum quality
- Score delta is a natural observable to expose in observability/tracing

**Cons:**
- Threshold value requires empirical calibration (0.05 suggested, but model-dependent)
- Slightly harder to test (non-deterministic call count in production)
- Possible local minimum trap: if early rounds produce a good-but-not-best answer, the
  loop may exit before discovering a much better trajectory

---

### Option C — Tournament Bracket (M Alternatives Compete Pairwise, Losers Eliminated)

**Description:**
Instead of scoring each alternative independently per round, run **pairwise comparisons**
between alternatives and eliminate losers. For M=4:
```
Round 1: [a1 vs a2] → winner w1; [a3 vs a4] → winner w2
Round 2: [w1 vs w2] → champion
```
The final champion becomes the seed for the next outer round. Inner tournament has
`log2(M)` depth; outer loop runs N rounds of tournaments.

**Total LLM calls:** N × (M + (M-1)) = N × (2M-1) for inner tournament; plus M
alternatives each time. For N=3, M=4: 3 × 7 = 21 calls minimum vs Option A's 12.
With LLM-as-judge for pairwise, cost doubles further.

**Implementation complexity:** High. Binary tournament tree, power-of-2 constraint on
M, pairwise judge, recursion within a round. Significantly harder to implement and test.

| Dimension | Assessment |
|---|---|
| Quality | Potentially highest (head-to-head comparison is more discriminating than absolute scores) |
| Token cost | Higher than A or B; pairwise comparison costs extra calls |
| Latency | Similar to A but with inner tournament adds depth |
| Complexity | High; tournament tree, bracket management, edge cases for odd M |
| Testability | Low; complex structure, many edge cases |
| Risk | High; complex failure modes, M must be power-of-2 or padded |

**Pros:**
- Pairwise relative comparison is more accurate than individual absolute scoring
- Natural quality ceiling higher than simple tournament when alternatives are close in score
- Results in diverse exploration of solution space at each level

**Cons:**
- `O(N × M × M)` LLM calls (once pairwise comparison added): unsustainable for large M
- M must be a power of 2 for clean brackets (complicates API)
- Evaluation bias is amplified: LLM-as-judge scoring its own outputs pairwise
- Significant implementation complexity; high risk for V1
- Not meaningfully better than Option A/B for typical agent task prompts

---

### Option D (Secondary) — Parallel Rounds (All M×N Calls in Parallel)

**Description:**
Issue all N×M LLM calls concurrently using `asyncio.gather`, then run scoring in a
single pass. No sequential round structure — purely parallel.

**Total LLM calls:** N × M simultaneously.

**Assessment:** This destroys the recursive "seed from prior best" structure that is
the core value of CoRT. Round N cannot use the winner of round N-1 as seed. This is
effectively batch generation, not recursive refinement. **Eliminated.**

## 6. Options: EvaluationEngine

### Option E1 — LLM-as-Judge (Secondary LLM Scores Each Chain)

**Description:**
Each generated chain is passed to a **judge call** — a second LLM prompt that asks
the model to score the chain on each rubric axis (0–10) and return structured JSON.
The scorer prompt includes the original question, the chain to score, and the rubric
definition. A secondary, smaller model (or same model at temperature 0) is used.

```python
judge_prompt = f"""
Original question: {original_prompt}

Candidate answer:
{chain}

Score this answer on three axes from 0–10:
- Correctness: factual accuracy and logical consistency
- Completeness: covers all aspects of the original question
- Reasoning_depth: shows step-by-step derivation

Return JSON only: {{"correctness": X, "completeness": Y, "reasoning_depth": Z}}
"""
```

**Request flow:** For each of M alternatives per round, 1 judge call is issued
(optionally in parallel with `asyncio.gather`). Total extra calls: N × M judge calls.

| Dimension | Assessment |
|---|---|
| Accuracy | Highest; semantic understanding of correctness |
| Cost | High; N×M extra LLM calls (doubles total) |
| Speed | Slow if sequential; acceptable if parallel |
| Reliability | Medium; depends on model's ability to self-evaluate; known bias |
| Testability | Medium; needs live mock LLM or fixtures; JSON parse failure modes |

**Pros:**
- Only approach that can reliably score **correctness** (factual accuracy)
- Scores structured JSON are easy to consume
- Works for any domain without domain-specific heuristics

**Cons:**
- Doubles token cost (N × M × 2 total calls)
- Strong **self-evaluation bias**: a model scoring its own outputs may consistently
  prefer outputs matching its own style (sycophancy amplification)
- JSON parse failures require error handling and retry logic
- Judge model quality matters; cheap models produce unreliable scores

---

### Option E2 — Rubric-Based Heuristics (No LLM Calls for Scoring)

**Description:**
Compute proxy scores using text analysis heuristics:
- **Correctness proxy**: absence of contradiction markers ("but wait", "actually no",
  "I was wrong"), sentence consistency (no opposing claims in same paragraph)
- **Completeness proxy**: word count vs median, coverage of prompt keywords,
  presence of a conclusion section
- **Reasoning-depth proxy**: count of reasoning markers ("because", "therefore",
  "since", "given that", "step N:", numbered lists), code block presence for code prompts

Each axis yields a 0–10 normalised score. Python `re` module + simple NLP patterns.
No external dependencies required beyond stdlib.

| Dimension | Assessment |
|---|---|
| Accuracy | Low–medium; proxies only, no semantic understanding |
| Cost | Zero extra LLM calls |
| Speed | <1ms per chain |
| Reliability | High; deterministic, no network failure modes |
| Testability | Highest; pure functions, no mocks needed |

**Pros:**
- Zero additional LLM cost — critical for V1 budget
- Completely deterministic → unit tests are trivially simple
- Fast (microseconds), no latency impact on the main loop
- No external dependencies; stdlib only (`re`, `collections`)
- Correctness proxy (contradiction detection) is surprisingly useful for catching
  logical self-contradictions even without semantic reasoning

**Cons:**
- Cannot reliably score factual accuracy of answer content
- Word-count based completeness proxy penalises concise but complete answers
- Reasoning-depth proxy is easily gamed by verbose filler text
- Keyword-based heuristics are domain-specific; a code answer differs from a research answer

---

### Option E3 — Ensemble (Heuristic Pre-Score + Optional LLM Final-Score)

**Description:**
Two-tier scoring pipeline:
1. **Tier 1 (always runs)**: heuristic scorer (E2) produces a preliminary score for
   each chain in <1ms.
2. **Tier 2 (optional, enabled by `use_llm_judge=True`)**: LLM judge (E1) re-scores
   only the **top K** chains from Tier 1 (default K=2) to refine rankings among the
   top candidates. Losers (below Tier-1 threshold) are eliminated without LLM cost.

This approach uses heuristics to **prune** the candidate space and LLM judge to
**discriminate** among survivors. For M=3, only K=2 LLM judge calls are needed per
round instead of M=3 — a 33% cost reduction vs full LLM judge.

```python
# Tier 1 fast pass
scored = [(heuristic_score(c), c) for c in alternatives]
scored.sort(reverse=True)

# Tier 2 optional refinement of top K
if use_llm_judge:
    top_k = scored[:k]
    llm_scores = await asyncio.gather(*[llm_judge(c) for _, c in top_k])
    # blend: final = 0.4 * heuristic + 0.6 * llm_judge
    ...

best = select_winner(scored)
```

| Dimension | Assessment |
|---|---|
| Accuracy | Medium–High; LLM judge only on top-K eliminates noise from weak candidates |
| Cost | Configurable: zero extra cost (use_llm_judge=False) to K×N×M (Full LLM mode) |
| Speed | Fast in heuristic mode; controlled latency in ensemble mode |
| Reliability | Medium; graceful fallback to heuristics if judge call fails |
| Testability | Medium; must test both paths; judge mock optional |

**Pros:**
- Configurable from zero-cost (V1) to full-accuracy (production) without API change
- Heuristic eliminates clearly-weak chains before expensive judge calls
- Natural upgrade path: ship V1 with `use_llm_judge=False`, enable later
- Graceful degradation: judge failures fall back to heuristic silently
- Blended scores are more robust than either scorer alone

**Cons:**
- More complex than E2 alone
- Heuristic pre-ranking must be reasonably correlated with truth for pruning to work
- Two score sources require blending calibration (weights 0.4/0.6 need empirical validation)
- Top-K selection adds a hyperparameter (`k`) to tune

## 7. PyAgent Integration Points

### 7.1 Module Layout

The project spec targets `src/core/reasoning/` as the new module. This is correct:
it is distinct from `src/cort/` (prior art tree module) and follows the
`src/core/<domain>/` pattern established by `src/core/providers/`.

```
src/core/reasoning/
├── __init__.py          # exports CortCore, CortAgent, EvaluationEngine
├── CortCore.py          # recursive reasoning loop
├── CortAgent.py         # agent wrapper / mixin
└── EvaluationEngine.py  # scoring engine
```

### 7.2 Mixin vs Standalone Agent

**Option I1 — Pure Mixin (multiple inheritance only):**
```python
class MyCortAgent(CortAgent, SomeOtherAgent):
    ...
```
CortAgent adds `async reason(prompt, **kwargs)` to any `BaseAgent` subclass via
Python multiple inheritance. No `__init__` modification required on the host class.

**Option I2 — Standalone Agent with optional mixin helper:**
```python
class MyCortAgent(BaseAgent):
    cort = CortCore(llm_callable=..., rounds=3, alternatives=3)
    async def run(self, task):
        return await self.cort.reason(task.prompt)
```
CortAgent is a factory/wrapper that any agent can instantiate as a component.

**Option I3 — Both: CortAgent as stand-alone AND as mixin target:**
```python
# Standalone use:
agent = CortAgent(llm_callable=adapter.complete, rounds=3)
result = await agent.reason(prompt)

# Mixin use:
class SmartCoder(CortAgent, BaseAgent):
    ...
```

**Assessment:** Option I3 is the recommended pattern. It aligns with the project spec
("mixin-based" architecture) while allowing standalone usage without requiring multiple
inheritance. `CortAgent` should internally delegate to `CortCore` (maintaining the
Core/Agent separation mandated by `copilot-instructions.md`). The mixin adds
`reason()` and optionally overrides `run()` to route through CoRT.

The key rule from `copilot-instructions.md`:
> *Core/Agent Separation: Domain logic must reside in a separate `*Core` class to
> allow for performance optimization (e.g., Rust FFI bridge). The Agent class should
> only handle orchestration, AI prompting, and state.*

This means:
- `CortCore.py` — pure domain logic: the recursive loop, alternative generation, and
  score ranking
- `CortAgent.py` — orchestration: reads agent manifest, opens ContextTransaction,
  calls `CortCore.reason()`, handles errors, logs via observability

### 7.3 ContextTransaction Usage

Each call to `CortAgent.reason(prompt)` must be wrapped in a `ContextTransaction`
to prevent infinite recursive re-entry:

```python
async def reason(self, prompt: str) -> str:
    ctx_id = f"cort:{self.agent_id}:{uuid.uuid4().hex[:8]}"
    async with ContextTransaction(ctx_id):
        return await self._core.reason(prompt)
```

The `context_id` is scoped to the agent instance (`self.agent_id`) to allow
**parallel CoRT agents** to operate concurrently without conflicting. Each
invocation gets a unique suffix so the same agent can process sequential tasks.

**Recursion guard coverage:**
- `ContextTransaction.__aenter__` raises `RecursionGuardError` if the same
  `context_id` is already active in this asyncio task
- Per-task isolation via `contextvars.ContextVar` means two concurrent agents in
  different asyncio tasks do NOT block each other
- Depth guard for nested CoRT (e.g., a CoRT agent whose tool calls another CoRT
  agent) is handled transparently by the stack-based parent tracking

**Important:** The context_id scheme above uses a UUID suffix per invocation, so
the recursion guard only fires if the exact same UUID is re-entered (which would
require a bug in the calling code). If the intent is to prevent the *same agent*
re-entering CoRT *from within a CoRT tool call*, the context_id should be stable:
`f"cort:{self.agent_id}"` without the UUID suffix. The choice depends on whether
we want strict singleton-per-agent or just loop-break guard. **Recommendation:**
use stable `f"cort:{self.agent_id}"` so any re-entry from within a running CoRT
session is blocked.

### 7.4 MemoryTransaction for Caching

Two distinct caching use-cases:

**A. Inter-round state (required):** The winning chain from round N is used as seed
for round N+1. This is transient state local to one `reason()` invocation. A simple
local variable (`best_chain: str`) is sufficient — no MemoryTransaction needed.

**B. Prompt-level caching (optional, V2):** Cache the final result of a CoRT run
keyed by `hash(prompt + parameters)` so identical prompts don't re-run the full loop.
This would use `MemoryTransaction` with key `f"cort_cache:{prompt_hash}"`.
**Recommendation for V1:** Skip caching entirely. Add in V2 once the pipeline is stable.

### 7.5 `src/core/base/mixins/` Directory

The instructions mention `src/core/base/mixins/` as the pattern location, but the
directory does not exist. The project spec says to implement CoRT at
`src/core/reasoning/` — this is the correct approach. **Do not create
`src/core/base/mixins/`** as part of this project; that is a broader infrastructure
task outside scope. CoRT lives in `src/core/reasoning/` and acts as its own mixin.

## 8. LLM Backend Compatibility

### 8.1 Abstract LLM Protocol

The existing `FlmChatAdapter` wraps an OpenAI-compatible client. CortCore must not
hard-depend on `FlmChatAdapter`. Instead, it should accept a **typed callable**
that produces a string response given messages:

```python
from typing import Protocol, Any

class LlmCallable(Protocol):
    async def __call__(
        self,
        messages: list[dict[str, str]],
        *,
        temperature: float,
        **kwargs: Any,
    ) -> str:
        """Call the LLM and return the response text."""
        ...
```

Any backend (vLLM, OpenAI, Ollama, local FLM) can wrap itself to match this protocol.
`FlmChatAdapter` would expose an async method:

```python
async def complete(
    self,
    messages: list[dict[str, str]],
    *,
    temperature: float = 0.7,
    **kwargs: Any,
) -> str:
    response = self._create_client().chat.completions.create(
        model=self.config.default_model,
        messages=messages,
        temperature=temperature,
        **kwargs,
    )
    return response.choices[0].message.content
```

**CortCore constructor signature:**
```python
def __init__(
    self,
    llm: LlmCallable,
    *,
    rounds: int = 3,
    alternatives: int = 3,
    base_temp: float = 0.7,
    temp_step: float = 0.1,
    max_temp: float = 1.2,
    evaluation_engine: EvaluationEngine | None = None,
) -> None:
```

### 8.2 Temperature Variation Across Alternatives

Generate M alternatives in parallel via `asyncio.gather`:

```python
temperatures = [
    min(base_temp + i * temp_step, max_temp)
    for i in range(alternatives)
]
tasks = [
    self._llm(messages, temperature=t)
    for t in temperatures
]
chains = await asyncio.gather(*tasks)
```

This is safe because each call is independent. Parallel generation reduces latency
from serial M×(avg_latency) to approximately 1×(avg_latency) + coordination overhead.

**Safety constraints on temperature:**
- Temperature 0.0 is allowed (deterministic/greedy); useful for judge calls in E1
- Temperature > 1.5 may degrade grammatical coherence; clamp at `max_temp`
- `base_temp` of 0.7 is a reasonable default for balanced diversity vs quality
- For code generation tasks, lower `base_temp=0.3` with `temp_step=0.1` is recommended
  (code rewards precision over diversity)

### 8.3 Backend Registration

CortCore does not need a backend registry. The caller (CortAgent or direct user)
provides the callable. This is simpler than a registry and more testable (just inject
a mock callable in tests).

## 9. Risk Analysis

### 9.1 Recursive Cost Explosion (HIGH likelihood / HIGH impact)

**Risk:** N × M LLM calls + optional N × M judge calls. With large N and M, and/or
nested CoRT agents calling each other, token cost can reach prohibitive levels quickly.

**Prevention measures:**
1. Hard defaults: N=3, M=3 (9 total calls — acceptable for most tasks)
2. Configurable budget cap: `max_llm_calls` parameter; `CortCore` raises `BudgetError`
   if `rounds × alternatives` exceeds cap
3. ContextTransaction blocks re-entry, preventing nested CoRT agents from creating
   nested loops
4. Expose `estimated_calls(rounds, alternatives)` as a static method for callers to
   budget before starting

**Mitigation for V1:** Enforce maximum `rounds=5`, `alternatives=5` in constructor
validation. Any higher value raises `ValueError` at construction time (not runtime).

| Parameter | V1 Hard Cap |
|---|---|
| `rounds` | max 5 |
| `alternatives` | max 5 |
| Combined (N×M) | max 15 |

### 9.2 Context Window Overflow (MEDIUM likelihood / HIGH impact)

**Risk:** Each round appends the prior best chain as context for the next round.
After N rounds, the messages list includes the original prompt + N winning chains.
For long chains and N=3: possible to exceed 8K–16K token context window of smaller
models.

**Prevention measures:**
1. Truncate the seed chain if its length exceeds `max_seed_tokens` (default 2048 tokens)
2. Use only the winning chain's final answer section (strip reasoning prefix) as seed
3. Count tokens before issuing each call (simple `len(text.split())` proxy is sufficient
   for a guard; exact tokenization is not required)
4. Raise `ContextOverflowError` if estimated token count exceeds `context_limit`

**Mitigation for V1:** Pass only the last winning chain as seed (not the full history).
The messages list for round N is: `[system_prompt, user_prompt, assistant seed chain]`
— three messages maximum, not N accumulation.

### 9.3 Evaluation Bias — LLM Judging Its Own Outputs (MEDIUM likelihood / MEDIUM impact)

**Risk:** When the same model generates chains AND judges them, it may exhibit
sycophancy: preferring outputs that match its own style even if factually inferior.
In the LLM-as-judge configuration, this biases selection toward the model's
"comfortable" outputs, reducing true diversity.

**For V1 (heuristic scorer only):** NOT APPLICABLE — heuristics have no self-judgment
bias.

**For V2 (LLM judge enabled):**
1. Use a different model or temperature=0 for judge calls to reduce style bias
2. Randomise the order chains are presented to the judge (positional bias mitigation)
3. Use two-judge ensemble with aggregation for critical tasks

### 9.4 Infinite Loop Guard Failure (LOW likelihood / HIGH impact)

**Risk:** If `RecursionGuardError` is not raised when expected (e.g., unit test
catches too broadly or the active set is not properly restored on exception), a
CoRT agent could theoretically enter an infinite reasoning loop.

**Prevention measures:**
1. `ContextTransaction` uses `contextvars` (per-asyncio-task isolation) — the guard
   is sound as implemented
2. `RecursionGuardError` should propagate uncaught through `CortAgent.reason()` — it
   is a programming error, not a recoverable runtime error
3. Unit tests must cover: normal exit restores active set, exception exit restores
   active set (test `__aexit__` on exception path)
4. Outer `rounds` loop is a simple `for` loop — not recursive Python calls — so Python
   stack overflow is not a risk

### 9.5 Score Ties and Ranking Instability (LOW likelihood / LOW impact)

**Risk:** Two alternatives receive identical scores. Deterministic tie-breaking required
to avoid non-deterministic test failures.

**Prevention:** Sort with a stable sort strategy: primary key = composite score,
secondary key = alternative index (lower index = earlier generated = lower temperature = preferred).

### 9.6 JSON Parse Failure in LLM Judge (MEDIUM likelihood / LOW impact in V1)

**Risk:** LLM judge prompt asks for JSON; model may return malformed JSON, explanation
text, or partial structure.

**Mitigation:** Catch `json.JSONDecodeError`; fall back to heuristic score for that
chain. Log parse failure at WARNING level. This only applies to the optional E1/E3
judge path; V1 heuristics are not affected.

## 11. Recommendation

### 11.1 Recommended Recursive Loop: Option A — Fixed N Rounds

**Rationale:**
1. For V1, correctness and testability outweigh marginal cost savings from adaptive
   stopping. Option A is trivially provable correct — a for-loop with fixed iterations.
2. The adaptive variant (Option B) requires threshold calibration that depends on
   EvaluationEngine score ranges. Since the E2 heuristic scores may have different
   absolute ranges than LLM judge scores, threshold portability is uncertain.
3. Option C adds complexity disproportionate to its quality benefit for the task types
   CoRT targets in PyAgent (code generation, security analysis, reasoning tasks — not
   open-ended creative tasks where pairwise discrimination matters most).
4. The hard cap (`rounds ≤ 5`, `alternatives ≤ 5`, `N×M ≤ 15`) eliminates the cost
   explosion risk while keeping Option A's simplicity.

**Adaptive stopping (Option B) should be added in V2** as a constructor flag
`early_stop_threshold: float | None = None`. When None (default), behaves as Option A.
This lets V1 deliver cleanly and V2 iterate on top.

### 11.2 Recommended EvaluationEngine: E2 (Heuristics) with E3 Interface

**Rationale:**
1. E2 requires zero extra LLM calls, keeping V1's total cost at exactly N × M calls.
2. Pure-function heuristic scoring is trivially unit-testable without mocks.
3. The interface must be designed as E3 from the start: `EvaluationEngine` with
   an `async score(chain: str) -> float` method that accepts a `use_llm_judge` flag
   at construction time. V1 ignores the flag; V2 activates it.
4. The rubric axes (correctness, completeness, reasoning-depth) are meaningful even
   with heuristics:
   - **Correctness proxy**: word-level contradiction detection (negation + prior claim),
     absence of self-correction phrases ("I was wrong", "let me reconsider"),
     assertion density (substantive claims per sentence)
   - **Completeness proxy**: prompt keyword recall (what fraction of nouns/verbs in
     the prompt appear in the response), sentence count normalised to prompt complexity
   - **Reasoning-depth proxy**: logical connective density ("because", "therefore",
     "since"), ordered list presence (step-by-step), code block presence (for code tasks)

### 11.3 Architecture Summary

```
CortAgent (mixin/standalone)
  └── CortCore (domain logic)
        ├── LlmCallable (protocol, backend-agnostic)
        ├── EvaluationEngine (heuristic V1, LLM-judge V2)
        └── ContextTransaction (recursion guard)
```

**CortAgent** extends `BaseAgent`, adding:
- `async reason(prompt: str, **kwargs) -> str` — the public entry point
- Opens `ContextTransaction(f"cort:{self.agent_id}")`
- Delegates to `self._core.reason(prompt, **kwargs)`
- Optionally overrides `run(task)` to route task prompts through `reason()`

**CortCore.reason()** implements:
```
best_chain = ""
for round_idx in range(self.rounds):
    messages = build_messages(original_prompt, best_chain, round_idx)
    temps = [min(base_temp + i*temp_step, max_temp) for i in range(alternatives)]
    chains = await asyncio.gather(*[llm(messages, temperature=t) for t in temps])
    scored = [(await engine.score(c), c) for c in chains]  # parallel in V2
    best_chain = max(scored, key=lambda x: x[0])[1]
return best_chain
```

**EvaluationEngine.score()** implements:
```python
async def score(self, chain: str, *, original_prompt: str = "") -> float:
    c_score = self._correctness_score(chain)
    p_score = self._completeness_score(chain, original_prompt)
    r_score = self._reasoning_depth_score(chain)
    return (self.w_correctness * c_score
           + self.w_completeness * p_score
           + self.w_reasoning * r_score)
```

## 12. Open Questions for @3design

1. **LlmCallable interface**: Should `CortCore` accept the callable directly, or
   should it accept a `FlmChatAdapter` instance and call a specific method? The
   Protocol approach (recommended above) is more flexible but adds an interface
   contract. Confirm the callable signature before implementation.

2. **run() override**: Should `CortAgent.run(task)` automatically route through
   `reason()`, or should it be opt-in (the host agent explictly calls `self.reason()`)? 
   Auto-routing is convenient but may surprise developers who don't want all `run()` 
   calls to go through the full CoRT loop.

3. **Scoring of scoring calls**: The heuristic scorer calls `score()` synchronously,
   but the interface signature is `async`. Should V1 score all M chains in parallel
   (`asyncio.gather`) or sequentially? Given E2 heuristics are <1ms, sequential is
   fine, but parallel is the correct pattern for when E3 adds async LLM judge.
   **Recommendation:** use parallel gather form even in V1 to avoid a breaking change later.

4. **Losing chain log**: Project doc asks whether losing chains should be logged for
   fine-tuning datasets. Decision: yes, log at DEBUG level with structured fields
   `{round, rank, score, chain_preview[:200]}`. This is zero-cost (log level) but
   enables dataset collection.

5. **Error handling for LLM failures mid-round**: If one of the M parallel calls
   fails (timeout, network error), should CoRT (a) fail the entire round, (b) proceed
   with the remaining M-1 alternatives, or (c) retry the failed call? Recommend (b):
   minimum 1 alternative required, partial failures acceptable. `CortCore` should
   propagate `ValueError("No alternatives returned")` only if ALL M calls fail.

6. **`context_id` stability**: Confirm whether `ContextTransaction` context_id for
   CoRT should be per-agent-instance (stable) or per-invocation (unique). Recommended
   above: per-agent-instance (`f"cort:{self.agent_id}"`). This prevents the same agent
   from being CoRT-called recursively within a tool chain, but allows the same agent
   to process sequential tasks (since ContextTransaction is exited/re-entered).

7. **Module namespace**: `src/cort/__init__.py` (prior art) and `src/core/reasoning/`
   (new target) are distinct paths. Should `src/cort/` be deprecated or remains as-is?
   Recommended: leave `src/cort/` as-is (it's in scope of a different project); the
   new module is wholly independent.
