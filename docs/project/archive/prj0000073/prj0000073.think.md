## Key Design Observations

### Authentication model
- Two credentials: `X-API-Key` header / `api_key` query param (REST/WS) OR `Authorization: Bearer <JWT>` / `token` query param
- JWT algorithm: HS256, secret from `PYAGENT_JWT_SECRET`
- **DEV_MODE**: when neither `PYAGENT_API_KEY` nor `PYAGENT_JWT_SECRET` is set → all requests pass, one-time WARNING
- WS auth failure: close code 4401 (not standard HTTP 401 — important for client handling)

### WebSocket E2E encryption
1. Server generates ephemeral X25519 keypair
2. Server sends public key as base64 text frame
3. Client sends its public key as base64 text frame
4. Both derive 32-byte session key via ECDH
5. All subsequent frames: `base64(AES-256-GCM(payload))`

### Rate limiting
- Token bucket per client IP, default 60 req / 60 s (env: `RATE_LIMIT_REQUESTS`, `RATE_LIMIT_WINDOW`)
- `/health` is exempt
- On breach: 429 with `Retry-After: <window>` header
- `X-Forwarded-For` honoured for correct IP detection behind proxies

### Versioning middleware
- All `/api/v1/` responses: `X-API-Version: 1`
- All `/api/` (non-v1) responses: `Deprecation: true`, `Link: <v1-path>; rel="successor-version"`

## Prior Art / Alternatives Explored

### Option A — Auto-generate from OpenAPI JSON
FastAPI produces `/openapi.json` automatically. Tools like `redoc-cli` or `widdershins` can convert to markdown.
- **Pro**: always in sync with code
- **Con**: schema-centric, loses narrative explanation; no WebSocket section; generated output is hard to curate

### Option B — Hand-crafted static markdown
Write each endpoint section by hand, based on code reading.
- **Pro**: full narrative control; can include Mermaid sequence diagrams; easier to read
- **Con**: can drift from code changes over time

### Option C — Hybrid (chosen)
Use the FastAPI `/openapi.json` schema as the source of truth for request/response schemas (copy exact field
names and types), but write the reference pages by hand around those schemas. Add non-schema narrative
(auth guide, WebSocket protocol, error codes) manually.
- **Pro**: accuracy + readability + Mermaid diagrams
- **Con**: moderate maintenance burden (mitigated by @6code updating docs on each endpoint change)

**Decision: Option C (Hybrid hand-crafted).**
