## Selected Approach

Hybrid hand-crafted markdown (see think.md Option C). Five files under `docs/api/`:

```
docs/api/
  index.md            # Overview, quick-start table of contents, base URL, versioning note
  authentication.md   # API key, JWT, DEV_MODE, WebSocket auth, security recommendations
  rest-endpoints.md   # All REST endpoints, grouped by domain, request/response schemas
  websocket.md        # WebSocket protocol, Mermaid sequence diagram, message JSON schema
  errors.md           # HTTP status codes, WebSocket close codes, rate limiting
```

---

## File Design

### `index.md`
- Short intro paragraph: what the API does, who it is for
- Base URL block (dev/prod)
- Versioning table (v1 vs legacy)
- Response headers overview (X-API-Version, Deprecation, X-Correlation-ID, Retry-After)
- Links to each of the four detail pages

### `authentication.md`
- Two auth methods: API Key and JWT HS256
- Auth method comparison table
- Code examples: HTTP headers (REST) and query params (WebSocket)
- DEV_MODE warning with exact env var names
- Security recommendations

### `rest-endpoints.md`
- Group by domain:
  - Health
  - Metrics
  - Agent Ops (agent-log, agent-doc, agent-memory)
  - Projects
  - Pipeline
  - Watchdog
  - Plugins
- Each endpoint section contains:
  - Method + path
  - Auth required badge
  - Path/query/body parameters table
  - Request body example (JSON)
  - Response body example (JSON)
  - HTTP status codes

### `websocket.md`
- Connection URL
- Auth (query params)
- Mermaid sequence diagram for full handshake + message lifecycle
- Message format (JSON fields)
- Disconnect codes

### `errors.md`
- HTTP error table: 400, 401, 403, 404, 422, 429, 500
- WebSocket close code table: 4401, 1011
- Rate limiting detail (token bucket, env vars, Retry-After header)
- Example error response bodies (JSON)

---

## Markdown Standards

- ATX headings only (# / ## / ###)
- Fenced code blocks with language tag (`json`, `http`, `mermaid`)
- Tables for parameter lists and status codes
- No HTML tags (renders in MkDocs and GitHub both)
- Max line length: 120 characters (repo standard)
- No trailing whitespace
- Files start with `# Title` heading, second line is _metadata italic_

---

## Integration with MkDocs

`mkdocs.yml` already has a `docs/` source. The new files slot in as-is.
No changes to `mkdocs.yml` are required unless the nav section needs updating (deferred to @6code).
