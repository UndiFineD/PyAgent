# prj0000076 — future-ideas-kanban: Think Phase

_Status: DONE_
_Owner: @2think | Updated: 2026-03-26_

## Methodology

This audit performed a structured, read-only review of all 9 designated areas of the PyAgent repository.
Each area was explored via directory tree traversal, reading of key configuration and source files, and
cross-referencing with existing kanban entries (prj0000071–prj0000075) to avoid duplication.
Findings were filtered through a SWOT lens: Weaknesses and Threats receive higher urgency scores;
Opportunities and Strengths inform roadmap positioning. The 38 ideas surfaced below cover every area
and are ordered by SWOT-adjusted priority.

## P1/P2 SWOT Rationale

**idea-001 — private-key-in-repo (T / P1)**
A committed private key (`rust_core/2026-03-11-keys.priv`) is an active **Threat** to the repository. Anyone
with read access can clone the key and decrypt any data protected by it. The fact that it has been in the tree
for what appears to be weeks makes this critical—rotation must happen before any future public exposure or
contribution. A pre-commit `gitleaks` / `detect-secrets` hook is the structural fix to prevent recurrence.

**idea-002 — missing-compose-dockerfile (W / P1)**
`deploy/compose.yaml` is the canonical single-container deployment path and it references
`src/infrastructure/docker/Dockerfile` which simply does not exist. This is a **Weakness** that makes the
"getting started" experience broken for any new contributor who tries `docker compose up`. The Dockerfile must
be created (or the compose file corrected) to unblock all deployment work.

**idea-003 — mypy-strict-enforcement (W / P1)**
`mypy.ini` with `ignore_errors = True` renders the entire type annotation surface of `src/` decorative only.
This is a **Weakness** that undermines the architectural guidelines which call for PascalCase, UUIDs, and
async typing. Without type enforcement, regressions from incorrect parameter types or return types are
impossible to catch statically, which is especially dangerous in async swarm orchestration paths.

**idea-004 — quality-workflow-branch-trigger (W / P2)**
The `quality.yml` gate triggers on `prj0000026-*` which is a long-merged historic branch. PRs from any
other project branch skip the ruff lint, mypy, and coverage checks entirely. This **Weakness** means the
quality gate is functionally absent for all active development, undermining the entire CI investment.

**idea-005 — rust-ci-workflow (W / P2)**
`rust_core/` is the performance backbone of PyAgent (30+ Rust modules, PyO3 FFI). The absence of a
dedicated Rust CI job means clippy warnings accumulate (`clippy_out.txt` is present in the tree, suggesting
manual runs), `cargo test` regressions are invisible, and `cargo audit` never runs. This is a critical
**Weakness** for a Rust-accelerated system.

**idea-006 — codeql-ci-integration (O / P2)**
The repository already invested in custom CodeQL queries (`codeql/python/`, `codeql/rust/`). Not executing
them in CI wastes this existing **Opportunity**—static security analysis for injection, path traversal, and
unsafe Rust patterns could be automated at zero additional tooling cost.

**idea-007 — security-scanning-ci (T / P2)**
A `pip_audit_results.json` file committed manually (March 2026) indicates the team monitors vulnerabilities,
but this is a point-in-time snapshot. Without automated scanning per-push, new CVEs in the 30+ dependencies
represent an ongoing **Threat** to production deployments. Integrating pip-audit into CI closes this gap.

**idea-008 — coverage-minimum-enforcement (W / P2)**
A `--cov-fail-under=1` threshold is operationally equivalent to no coverage gate. With 183 test files and 101
source files, the infrastructure is in place—the **Weakness** is that no baseline prevents gradual erosion.
Setting 70% now and incrementing quarterly builds confidence systematically.

**idea-009 — requirements-ci-deduplication (W / P2)**
Duplicate lines in `requirements-ci.txt` (`pytest-xdist>=3.6` appears twice) indicate that dependency files
lack structural validation. This **Weakness** is minor in isolation but signals that no automated check
catches drift, duplicates, or conflicts before they propagate.

**idea-010 — docker-compose-consolidation (W / P2)**
Having both `deploy/compose.yaml` (GPU/Ollama stack) and `deploy/docker-compose.yaml` (fleet topology) with
no shared services or documentation creates **Weakness** through ambiguity—contributors don't know which
compose file to use. One parameterised file with `profiles:` would serve both cases.

**idea-011 — stub-module-elimination (W / P2)**
Six modules exist as empty `__init__.py` stubs. They bloat import paths, confuse contributors, and add
surface area to `pytest --collect-all` without test coverage. Removing or properly scaffolding these
**Weaknesses** would shrink the apparent complexity of `src/` significantly.

**idea-012 — dependabot-renovate (O / P2)**
No automated dependency update bot is configured despite having 30+ Python deps, 35+ Rust crates, and
npm packages. This low-cost **Opportunity** closes the gap between manual `pip-audit` snapshots and
continuous dependency freshness.

**idea-013 — backend-health-check-endpoint (W / P2)**
A FastAPI backend without a `/health` endpoint cannot be effectively monitored, load-balanced, or managed
by any container orchestrator or uptime check. This **Weakness** blocks production-grade deployment.

**idea-014 — pyproject-requirements-sync (W / P2)**
Two divergent install paths (`pip install -e .` via pyproject vs. `pip install -r requirements.txt`)
produce different environments with different package versions, causing "works on my machine" failures.
This **Weakness** must be resolved by making `requirements.txt` generated from `pyproject.toml`.
